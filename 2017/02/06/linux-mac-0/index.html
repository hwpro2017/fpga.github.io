Linux系统Vivado license问题 HostID="000000000000"

发表于 2017-02-06 | 分类于 工具 | | 次

Linux下，Vivado无法生成bit文件；
Linux下，Vivado提示license，而同样license在Win系统下缺没问题；
Linux下，Vivado License Manager (VLM) 提示 HostID为 “000000000000”；

有没有遇到过这种情况的玩家呢？
下面就解决这个问题。

问题检查

出现上述这个问题，一般是由于玩家所使用的Linux电脑或服务器中网卡名称采用了新的一致性命名方式biosdevname；

传统的网口名称eth0变成了类似下面这些名字

  • em1
  • em2
  • em3
  • em4
  • lo
  • p3p1
  • p3p1_0
  • p3p1_1
  • p3p2
  • p3p2_0
  • p3p2_1
  • p4p1

如果确实是这样情况的玩家，就可以继续往下看了。

好好的eth，为啥要改名呢？

运行 Linux 的系统一贯将以太网设备命名为 ethX。桌面可能有一个命名为 eth0的以太网端口。如果仅有一个网络端口，这样毫无问题，但如果四个以太网端口该怎么办？
它们被命名为 eth0、eth1、eth2、eth3，分别对应机箱背面的标签 1、2、3、4。
某些时候确实如此。除了从 0 开始，而不是从 1 开始导致的明显命名混乱外，还可能出现其他混乱情况，如每个端口在每次启用时可能无法获得相同的名称，此外它们的命名顺序可能是任意的。如果将网卡添加到 PCI 插槽，情况还会更糟，主板上的端口和新添加卡上的端口命名可能相互混杂。

于是乎，DELL想了一招

运行biosdevname，这是 Dell 开发的工具，旨在为网络设备的命名理清思路（并且保持一致性！）。Biosdevname 是一款 udev 帮助程序，可根据系统 BIOS 提供的信息对网络接口进行重命名。

新的命名规则如下

em[1-N] 表示主板（嵌入式）NIC（# 对应机箱标签）
pci# 表示 PCI 插槽中的卡，端口 1 至 N
NPAR & SR-IOV 设备增加后缀 _，从 0 至 N，取决于各端口可用的分区或虚拟功能的数量。其他 Linux 命名规则，如 . 和 : 后缀，保持不变且仍然适用。由此，Linux 网络接口名称与外部可见的网络端口（RJ-45 机架）之间建立了明确的对照。

相关详情可参考

“Linux一致性网络设备命名规范白皮书”

问题原因

这些和咱们的问题有啥关系呢？
问题的关键来了：

赛灵思的License管理工具（VLM）是基于Flexlm工具是无法识别这些新名字的！

解决方案

既然找到问题原因了，那就可以找办法解决了。
既然不识别新的一致性网络设备名称，那咱就给他改回去呗。

  • cd /etc/default
  • sudo vi grub
  • 在GRUB_CMDLINE_LINUX中添加: “net.ifnames=0 biosdevname=0”
  • 保存退出：:wq
  • grub2-mkconfig -o /boot/grub2/grub.cfg
  • cd /etc/sysconfig/network-scripts
  • sudo mv ifcfg-enp0s3 ifcfg-eth0
  • sudo vi ifcfg-eth0
  • 将 enp0s3（或类似的biosdevname新名字）改为 eth0
  • 保存退出：:wq
  • 重启后生效

生效后重新运行Vivado，应该就不会遇到之前的问题啦！
