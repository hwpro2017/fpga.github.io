Vivado上手教程2-基本硬件设计流程

发表于 2017-02-26 | 分类于 教程 ， ZYNQ | | 浏览 次

本文是ZYPI平台上的第二个实验。
本实验提供预先编写好的Verilog源代码。 实验中将使用Vivado进行Verilog设计仿真来测试模块功能并进行Verilog设计Debug；随后进行设计的综合以及实现，最终生成Bitstream，并将其下载到ZYPI板卡并验证硬件中的功能。 通过观察板卡现象，你会注意到这个设计的功能和Lab1的行为是相同的。也就是说软件中实现的相同功能可以在硬件中实现（在硬件中实现起来相对更复杂，但是有时这样做可以实质性的改进系统性能，我们会在后续继续讨论）。

实验介绍

在上一个实验中（ZYPI第一个实验，请点这里），我们使用了Zynq芯片的PS和PL侧。在ARM处理器上运行了一个软件来实现一个二进制计数器。在本实验中将实现相同功能的计数器，但使用完全不同的方法。 在本实验中，我们将只使用芯片的PL端。仅由硬件电路实现完全相同的功能。 使用Verilog HDL（硬件描述语言）描述硬件，通过Vivado综合器将Verilog文件转换为电路网表。 本实验将为用户提供一个良好的硬件设计流程启动指南，其中包括许多步骤：设计，仿真和验证以及实现等部分。

本实验中的设计方法是硬件工程师通常遵循的一个简化的实现硬件电路或IP的方法。 我们在上一个实验中使用的IP核也是以类似的方式实现，同样是由开发人员编写的Verilog HDL（或其他HDL）文件，并且封装成的即插即用IP核。

实验步骤

  1. 使用Vivado创建一个针对ZYPI平台的简单硬件HDL设计。
  2. 检查Vivado项目目录结构，并熟悉Vivado创建的目录以存储导入的源代码，新代码，报告和设计输出等相关文件。
  3. 对设计进行RTL分析，分析由Vivado为HDL设计生成的原理图。
  4. 在综合后和实现后中对设计进行硬件利用率分析。硬件利用率提供了在芯片内使用多少硬件资源的百分比以及相关详细报告。
  5. 介绍Vivado仿真器，执行行为仿真，编写Verilog测试台并且进行波形结果分析。

新建工程

按照之前的博文说明（点击这里查看），创建一个新的ZYPI工程。

添加源文件

在左侧Project Manager中选择Add Sources打开添加源文件窗口。

添加源文件窗口界面提供了如下几个选项：

  1. Add or Create Constraints（添加或者创建约束）；
  2. Add or Create Design Sources（添加或者创建设计源文件）；
  3. Add or Create Simulation Sources（添加或者创建仿真文件）；
  4. Add or Create DSP Sources（添加或者创建DSP源文件）；
  5. Add Existing Block Design Sources（添加已存在的块设计源文件）；
  6. Add Existing IP（添加已存在的IP）。

在添加源文件窗口中，单击绿色+符号，然后选择添加文件。按住Ctrl键选择Lab2.v，Freq_div.v，Direction_ctrl.v三个文件。这些文件为Lab2的硬件设计源文件。在“ 添加源文件 ”窗口中选中“ Copy Sources into project ”，确保源文件被复制进当前工程路径下。



完成源文件添加后，在Source窗口中可以看到刚才添加进来的源文件。

添加约束文件

通常，一个设计中的FPGA不会是独立使用的，FPGA一定会与其他外设、接口相连接，并且FPGA通常需要有外部时钟的接入。因此，FPGA设计需要在工具中指定对应的IO引脚位置以及输入时钟的信息，即需要用户对IO进行约束以及进行时钟周期等时序约束。在Vivado中，用户可通过I/O Planner进行IO约束，使用Timing Constraints Manager指定期望设计性能即进行设计的时序约束。


与添加设计源文件类似，通过相同的方式选择xdc文件添加进工程中。

在本实验中，约束文件的作用是为设计顶层的输入输出引脚进行了位置约束。


完成约束文件的添加后，在Source窗口中可以看到对应的约束文件。

RTL分析

Vivado工具提供对HDL源文件的RTL分析功能，通过RTL分析用户可以看到与源文件等效的RTL原理图，并在此基础上进行一些设计分析。

点击Open Elaborated Design来打开RTL分析结果，左侧会出现Vivado分析后的RTL原理图。


RTL分析后的原理图，原理图会和源文件的模块名称信号名称以及层级关系保持一致。

在展开模块后，可以看到整个设计的RTL原理图。该原理图是依据HDL描述生成的，根据该原理图可以查看设计是否达到要求并加以修改。在Vivado的设计过程中，用户可以在RTL分析、综合、实现阶段后打开设计原理图或设计网表来进行观察。在RTL描述与分析阶段后的原理图网表是由与FPGA底层部件无关的逻辑符号来体现的。可在后续过程中看到综合阶段后的原理图，在综合阶段后的原理图中的逻辑原件便是由FPGA中的底层部件来构建的了。

设计仿真

RTL Analysis能够非常方便的用于检查一些语法和逻辑错误，但是设计是否完全满足功能需求还需要通过仿真来验证。如果直接在FPGA芯片上测试，可能因为需要多次综合、实现而浪费大量时间，同时很难观察一些内部信号。而使用功能仿真则可以很快的测试出功能是否满足要求，并且避免多次综合、实现的过程。

Vivado工具集成的仿真工具功能十分强大，提供诸多功能

  1. 图形化波形显示窗口；
  2. 可通过工具栏按钮添加标尺、测量延时以及波形缩放；
  3. 总线信号可展开观察单根信号线；
  4. 可任意插入分隔线用于分隔相关信号组；
  5. 控制台窗口可打印testbench中的输出信息；
  6. 默认自动添加顶层信号的波形显示。

再次添加源文件，这次选择添加仿真激励文件TestBench

TestBench是指一个虚拟的针对待测设计的验证环境。TestBench用于向待测设计(Design Under Test)提供激励，通常也是HDL形式的文件。用户也可以在TestBench中实现对测试结果的自动检测。通常，一个TestBench测试激励文件中可能具有如下部分：

  1. 内部信号
  2. 待测设计的例化
  3. 时钟等输入激励信号
  4. 输出监视，结果比对
  5. 板级部件的仿真模型（如存储器等）


将设计目录下的仿真激励文件添加至工程中。


添加完成后，可以在Source窗口中看到设计激励，并且可以看到激励文件在设计顶层之上。

双击打开激励文件，可以看到在激励文件中完成了对设计顶层的例化以及相关激励信号的生成。

设计编译：设计综合与设计实现

设计综合

FPGA是基于查找表结构的，在FPGA中任何数字电路都是由查找表和寄存器等FPGA底层基本单元组成。设计综合的过程就是由FPGA综合工具将HDL、原理图或其他形式的源文件进行分析、推演出由FPGA芯片中底层基本单元表示的电路网表的过程。

综合选项

首先，用户在做Vivado的综合工作之前可以对Vivado工程设置中的一些和综合相关的设置做一些基本了解。


  1. tcl.pre 与tcl.post 选项用于指定Tcl钩子脚本（其将在综合前或后直接运行）；
  2. flatten_hierarchy: 将决定综合算法如何处理设计的层级关系；none: 不去将层级关系打平，保持在RTL描述中的设计层次关系；full: 完全将层次化的设计展开；rebuilt: 首先将层次关系展开，进行综合，然后按照原RTL再将设计的层级关系重建。
  3. gated_clock_conversion: 使能或不使能综合工具对门控时钟的自动转换；
  4. bufg: 控制工具推演使用BUFG资源的数量；
  5. fsm_extraction: 选择实现FSM的编码方式: one_hot、sequential、johnson、gray、或auto方式；
  6. keep_equivalent_registers: 防止有相同输入逻辑的寄存器合并；
  7. control_set_opt_threshold: 控制集阈值（在不超出阈值的情况下工具会做时钟使能优化）；
  8. no_lc: 当选中时，工具将不进行LUT的压缩；
  9. shreg_min_size: 推演SRL的阈值（超出后将推演出SRL）；
  10. max_bram: BRAM最大使用量（-1表示不做限制）；
  11. max_dsp：DSP的最大使用量。

设计实现

Vivado下的FPGA设计实现是指由FPGA实现工具将FPGA综合后的电路网表针对某个具体指定的器件以及相关物理与性能约束进行优化、布局、布线并生成最终可以下载到FPGA芯片配置文件的过程。

开始设计实现

实现过程

  1. 优化设计Opt Design : 针对所选器件，对逻辑设计进行优化，以便达到最优实现；
  2. 功耗优化设计Power Opt Design (可选):从降低功耗的角度，对逻辑设计进行优化；
  3. 布局设计Place Design(必选):将设计网表在所选器件上进行布局；
  4. 布局后功耗优化Post-Place Power Opt Design (可选):在布局之后的网表基础上优化功耗；
  5. 布局后物理优化Post-Place Phys Opt Design (可选):在布局之后的网表基础上进行物理优化，主要针对时序性能；
  6. 布线设计Route Design(必选): 在布局后的设计上，进行布线；
  7. 布线后物理优化Post-Route Phys Opt Design (可选):在布线后的设计上，参考布线后的设计延时，对逻辑、布局、布线等情况再次进行优化；

实现设置

同样的，Vivado设计实现部分也有一些相关的设置，我们打开工程设置中的实现设置窗口。可以看到，与设计综合类似，设计实现的每个子步骤都具有Tcl钩子脚本功能，用户可以使用Tcl脚本灵活的制定设计实现流程。此外，对于设计实现的每个子步骤，用户都可以去指定工具的Directive，即让工具按照设计者预期的目标进行设计实现结果的探索。

实现后的设计报告

Vivado实现过程包括逻辑与物理的转化、将网表布局布线到设备资源中。实现完成后我们可以查看布局布线的实际情况，核实是否符合设计要求。另外，Vivado工具在设计实现后可以提供诸多设计报告以供设计者查看。

其中包含以下一些经常使用的报告：

  1. Post Optimization DRC Report: 列出I/O DRC检查结果；
  2. Post Power Optimization DRC Report: 列出power DRC检查结果；
  3. Place and Route Log: 实现过程的记录文件；
  4. IO Report: 列出设计最终的IO情况；
  5. Clock Utilization Report: 详细列出时钟资源的使用情况；
  6. Utilization Report: 详细列出FPGA资源使用情况；
  7. Control Sets Report: 列出设计中控制信号的详细情况。
